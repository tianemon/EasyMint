# AI驱动开发需求设计原则（AORD v1.0）

我认为传统PRD（产品需求文档）是为人类团队设计的，而AI Agent时代需要一种新的需求组织方式。

目标不是让人看得舒服，而是让AI理解得准确、执行得稳定、迭代成本更低。

我把它称为：

> **AORD（AI-Oriented Requirement Design）**
>
> AI导向需求设计法

---

# 一、核心原则

## 原则1：不要描述功能，要描述工作流

很多人这样写需求：

```text
需要登录功能
需要搜索功能
需要收藏功能
需要推荐功能
```

这属于功能列表。

对于AI来说：

```text
登录
搜索
收藏
推荐
```

之间没有关系。

AI需要自己猜：

* 先登录还是先搜索？
* 收藏在哪里出现？
* 推荐依据什么生成？

容易产生幻觉和遗漏。

---

应该改成：

```text
用户进入网站

→ 搜索产品

→ 查看详情

→ 加入对比

→ 查看推荐

→ 收藏产品
```

形成完整业务流。

---

原因：

AI本质上是预测下一步。

工作流本身就提供了：

```text
状态
↓
动作
↓
结果
↓
下一状态
```

天然符合Agent推理方式。

---

# 原则2：从用户目标开始

不要从功能开始。

先写：

```text
用户想完成什么
```

例如：

```text
用户目标：

找到适合自己的手机
```

然后拆解：

```text
为了完成目标：

搜索手机
↓
筛选手机
↓
查看参数
↓
比较产品
↓
做出购买决策
```

---

错误：

```text
产品功能：

搜索
收藏
推荐
```

---

正确：

```text
用户目标：

选择手机

支持流程：

搜索
筛选
比较
推荐
收藏
```

---

原因：

AI会围绕目标进行规划。

而不是围绕零散功能进行拼接。

---

# 原则3：采用树状结构

不要写大段文字。

例如：

```text
产品详情页需要展示图片、视频、参数、规格、跑分数据、评测内容、价格走势……
```

AI容易漏。

---

改成：

```text
3. 产品详情页

3.1 基础信息

3.1.1 名称
3.1.2 品牌
3.1.3 型号

3.2 多媒体

3.2.1 图片
3.2.2 视频

3.3 参数

3.3.1 CPU
3.3.2 内存
3.3.3 电池

3.4 内容

3.4.1 评测
3.4.2 新闻
```

---

树状结构具有：

```text
完整性
可追踪性
可扩展性
```

AI最容易理解。

---

# 原则4：每个节点必须有输入和输出

很多需求缺少边界。

例如：

```text
搜索功能
```

AI不知道：

* 输入是什么
* 输出是什么

---

正确写法：

```text
2. 搜索产品

输入：

关键词

处理：

匹配产品库

输出：

产品列表
```

---

再进一步：

```text
2. 搜索产品

输入：

用户关键词

处理：

全文检索

输出：

产品列表

异常：

无结果
```

---

这样Agent几乎不会理解错。

---

# 原则5：每个功能都要定义状态变化

AI特别擅长状态机。

例如：

```text
收藏功能
```

改成：

```text
状态：

未收藏

动作：

点击收藏

结果：

已收藏

动作：

取消收藏

结果：

未收藏
```

---

实际上：

```text
状态机
```

比自然语言准确得多。

---

# 原则6：UI从工作流推导

不要直接设计页面。

先设计流程。

---

错误：

```text
需要首页
需要详情页
需要个人中心
```

---

正确：

```text
用户搜索产品
↓
结果页
↓
详情页
↓
对比页
```

然后得到：

```text
页面需求

首页
搜索结果页
详情页
对比页
```

---

这样页面不会凭空出现。

---

# 原则7：功能必须穷举

Agent最大的缺陷：

```text
默认补全
```

如果没写：

```text
删除
编辑
取消
异常
权限
```

Agent经常不做。

---

因此：

每个功能必须检查：

```text
创建
读取
更新
删除
```

即：

```text
CRUD
```

---

例如收藏：

```text
创建收藏

查看收藏

取消收藏
```

---

例如产品：

```text
新增产品

查看产品

修改产品

删除产品
```

---

# 原则8：显式写出异常流程

人会默认理解。

AI不会。

---

不要写：

```text
用户登录
```

---

应该写：

```text
登录

正常：

账号密码正确

结果：

进入系统

异常1：

密码错误

异常2：

账号不存在

异常3：

账号冻结
```

---

很多Agent遗漏问题都来自：

```text
异常流缺失
```

---

# 原则9：页面、数据、API同步设计

每个功能节点下面都写：

```text
UI

Data

API
```

例如：

```text
产品详情

UI

显示参数表

Data

Product

API

GET /product/{id}
```

---

Agent生成代码准确率会大幅提升。

---

# 原则10：先全局后局部

永远遵循：

```text
系统
↓
模块
↓
页面
↓
组件
↓
交互
↓
接口
```

---

不要直接让AI写：

```text
帮我做个产品详情页
```

因为：

```text
上下文缺失
```

---

应该先给：

```text
产品库系统
```

然后：

```text
产品模块
```

然后：

```text
产品详情页
```

最后：

```text
参数组件
```

---

# 原则11：全链路数据溯源

原则4要求每个节点有输入和输出——这是局部视角。

更进一步，关键数据在整个系统中的全局流转路径必须显式标注。

例如用户 Token：

```text
Token 全链路：

起点：LocalStorage.getItem('token')
↓
中转：AuthContext.State（内存）
↓
终点：每次请求注入 Axios Header Authorization
```

格式固定为三层：

```text
起点：数据从哪里读取
↓
中转：数据在哪个介质中持有（内存/缓存/状态树）
↓
终点：数据最终写入哪里（API/持久化/渲染）
```

原因：

Agent 在编写状态管理代码时，如果只看到"输入 token、输出验证结果"（局部），仍然可能在 Redux/Vuex/Zustand 的跨组件传递中写出逻辑错位。

当数据流被精确描述为 `起点 → 中转 → 终点` 的三段式时，Agent 在状态管理和 API 编排中几乎不可能写出数据来源或去向的歧义 bug。

---

# 推荐最终结构

一个完整AI项目需求文档建议如下：

```text
0. 项目目标

1. 用户角色

2. 用户工作流

3. 功能树

4. 页面树

5. 页面交互

6. 状态机

7. 数据模型

8. API设计

9. 权限设计

10. 异常流程

11. 测试场景
```

---

# 从需求到执行：原子化任务分发

结构化需求文档写完后，下一步是让 Agent 按序执行。

将功能树的每个叶子节点分配一个 `模块.序号` 编号：

```text
1.1 页面首次加载与状态路由
1.2 登录表单提交
1.3 登录结果处理
2.1 产品列表加载
2.2 搜索筛选
2.3 产品详情加载
```

Agent 按编号顺序执行，不跳跃：

- 逐任务报告进度（"功能 1.2 已就位"）
- 失败时精确定位到编号回滚
- 多 Agent 并行时按模块号分配

---

# 从需求到验证：闭环反馈

结构化需求文档的另一个价值：它天然定义了"完成"的标准。

每个功能节点对应的验收条件应当可自动化验证：

```text
功能 1.1 → 页面加载后 token 有效则重定向
功能 1.2 → 点击登录后按钮置灰，2s 内防抖
功能 2.1 → 产品列表 API 返回 200 且有数据
```

形成闭环：

```text
结构化需求 → Agent 编码 → 自动化测试 → 通过/驳回
                                    ↓ 驳回
                              错误信息自动反馈给 Agent
                                    ↓
                                修复 → 重测
```

当需求文档的结构足够严谨时，测试用例可以从需求直接生成，无需人工二次翻译。这就是 Harness Engineering 的核心理念——需求文档不仅是给 Agent 看的，更是 Agent 执行质量的自动化判定标准。

---

# AI时代需求文档的本质

传统PRD是：

```text
给人看
```

而AI需求文档应该是：

```text
给Agent执行
```

因此最重要的不是文笔，而是结构。

可以把整个原则浓缩成一句话：

> **先描述用户目标，再描述工作流；先构建树状结构，再补充状态、数据和异常；让每个节点都具备明确的输入、处理、输出和边界；再以数据溯源确保全链路无盲区，以原子编号驱动执行，以可验证标准形成闭环。**

如果做到这一点，你会发现 Agent 生成 UI、数据库、API、测试用例甚至后续迭代时，稳定性会比普通自然语言需求高出一个量级。
