# 基础操作封装层 — 设计方案

> 2026-06-05 整理。将之前在对话中关于「操作双轨合一」的分析整合为结构化方案文档。

---

## 1. 问题定义

### 1.1 当前架构的双轨问题

EasyMint 中存在两条独立的操作路径，各自走不同的业务约束：

```
用户点按钮 → IPC handlers → service 方法 → 执行业务逻辑 + 校验
   （有校验、有业务规则保护）

用户聊天说 → Mint → Bash/Write/Read 工具 → 直接操作系统
   （纯底层工具，业务规则只存在于 prompt 中的自然语言描述）
```

**核心矛盾**：业务规则有两种表达方式 —— 代码里的硬约束（按钮路径）和 prompt 里的软约束（聊天路径）。软约束可以被用户多磨几句绕过。

### 1.2 具体案例

| 场景 | 按钮路径 | 聊天路径 |
|------|---------|---------|
| 在 workspace 下创建项目 | `project-service.createProject()` 硬编码拒绝 | prompt 里写「不要在 workspace 下建项目」，但用户坚持时 Mint 可能妥协 |
| 创建会话 | 自动写入 sessions.json + 初始化 SDK session | Mint 可以跳过 sessions.json 记录，导致会话列表看不到 |
| 更新任务状态 | task.json 原子写 + 状态机校验 | Mint 直接 Write 覆盖 task.json，可能写错格式 |

### 1.3 期望目标

把两条轨道的业务逻辑合并为一条 —— **每个操作只有一种实现，无论从 UI 按钮还是 agent 聊天触发，走到同一段可执行代码**。

## 2. 架构方案

### 2.1 核心思路

引入 **Operations Layer**（操作封装层），位于现有 service 层之上或与之平级：

```
                        ┌──────────────────────────────┐
                        │      Operations Layer         │
                        │    （纯业务逻辑 + 校验 + 审计）   │
                        │                               │
                        │  createProject(name, path)    │
                        │  createSession(projId, title) │
                        │  updateTask(taskId, status)   │
                        │  writeProjectFile(proj, rel,  │
                        │    content)                   │
                        │  readProjectFile(proj, rel)   │
                        │  installDependency(proj, pkg) │
                        │  runProjectScript(proj, name) │
                        │  ...                          │
                        └──────────────┬───────────────┘
                                       │
                       ┌───────────────┴───────────────┐
                       │                               │
                  IPC Handlers                    MCP Server
                  (renderer→main)                (agent tools)
                       │                               │
                       ▼                               ▼
                 UI 按钮点击                     Mint 聊天调用
```

**关键设计决策**：
- Operations Layer 是一组纯逻辑方法，只包含业务规则和校验
- IPC Handlers 和 MCP Server 都是薄包装，只做参数格式转换
- 两条调用路径（UI 和聊天）最终执行同一段代码

### 2.2 Agent 侧接入方式

两种方案可选：

**方案 A：MCP Server（推荐）**

在 Electron 主进程内启动一个内建 MCP Server，把 operations 暴露为 MCP tools（如 `easymint_create_project`、`easymint_create_session`）。

- ✅ SDK 原生支持 MCP，agent 可以直接发现和调用
- ✅ 工具自带参数 schema，agent 知道参数类型和用途
- ✅ 可与现有 MCP 管理体系（`mcp-service.ts`）复用
- ✅ 工具执行在 Electron 进程内，天然可访问所有 service
- ❌ 需要在主进程内运行 MCP 协议服务（stdio transport）

**方案 B：SDK Hooks + `canUseTool`**

不提供新工具，而是在 agent 使用 Bash/Write 等底层工具时，通过 hook 拦截校验。

```ts
canUseTool: (toolName, input) => {
  if (toolName === 'Bash' && isDangerousOp(input)) {
    return { allowed: false, message: '请使用 createProject 操作' };
  }
}
```

- ✅ 实现简单，不需要新工具
- ❌ agent 先做错事再被拦截，体验差
- ❌ 只能禁止，无法引导 agent 用正确方式

**推荐 A + B 组合**：提供高层安全工具（A），同时在底层加护栏（B），双层防御。

### 2.3 与现有架构的契合

| 现有模块 | 如何衔接 |
|---------|---------|
| `project-service.ts` / `session-service.ts` / `file-service.ts` | Operations Layer 复用现有 service 方法，加标准化封装 |
| `mcp-service.ts`（MCP 管理） | 新增一个内建 MCP server，复用 mcp-service 的配置和启停机制 |
| `agent-service.ts`（`buildQueryOptions`） | 把内建 MCP server 注入 `options.mcpServers`，Mint 自动获得 operations 工具 |
| `agent-templates.ts` | 不同 agent 角色分配不同的 operations 工具集（如 Builder 没有 `createProject`） |
| `prompts.ts` | 加一条规则：「涉及项目管理操作时，优先使用 easymint 工具，不要用 Bash/Write 绕过」 |

## 3. 操作清单

分三批实现，第一批确定核心可行性。

### 3.1 第一批：核心操作（agent 最容易做错的）

| 操作 | 封装的业务规则 | 对应现有 service |
|------|--------------|-----------------|
| `createProject(name, path)` | workspace 检测、路径合法性、模板复制、注册到 projects.json | `project-service.ts` |
| `createSession(projectId, title)` | 项目存在性校验、SDK session 初始化、写入 sessions.json | `session-service.ts` + `agent-service.ts` |
| `updateTaskStatus(taskId, status)` | 状态机校验（pending→building→evaluating→done）、task.json 原子写 | `task-store.ts`（需主进程化） |
| `writeProjectFile(projPath, relPath, content)` | 路径必须在项目目录内、禁止覆盖 `.easymint/`、原子写 | `file-service.ts` |

### 3.2 第二批：辅助操作

| 操作 | 说明 |
|------|------|
| `readProjectFile(projPath, relPath)` | 限制读取范围在项目目录内 |
| `installDependency(projPath, package)` | 校验包名格式、记录到 package.json |
| `runProjectScript(projPath, scriptName)` | 只允许 package.json scripts 中定义的命令 |
| `deleteProject(projectId)` | 删除前二次确认、清理关联数据 |

### 3.3 第三批：扩展操作

| 操作 | 说明 |
|------|------|
| `gitCommit(projPath, message)` | 限制在项目目录内操作 |
| `createBranch(projPath, name)` | 分支命名规范校验 |
| `getProjectStatus(projPath)` | 统一的阶段检测逻辑（复用 `project-status-store`） |
| `addTask(projPath, task)` | 追加到 task.json，确保格式正确 |

## 4. 实现路径

### 4.1 阶段 1：接口定义 + 核心实现

1. 定义 `Operations` 接口 — 方法签名、参数类型、返回值、异常
2. 实现 `app/main/services/operations.ts` — 纯业务逻辑层，不关心调用方
3. 改造 IPC handlers — 改为调用 operations 方法而非直接调用 service

```
产出：
  app/main/services/operations.ts      — Operations Layer 实现
  app/shared/operations.types.ts       — 类型定义（main/renderer 共享）
```

### 4.2 阶段 2：Agent 侧接入

1. 实现内建 MCP server（`easymint-mcp`），注册 operations 为 tools
2. 在 `buildQueryOptions` 中注入该 MCP server
3. 更新 `prompts.ts` 规则，要求 Mint 优先使用 operations 工具

```
产出：
  app/main/services/operations-mcp.ts  — MCP Server 实现
  app/main/services/agent-service.ts   — 注入 MCP server 到 options
  app/shared/prompts.ts               — 更新行为规则
```

### 4.3 阶段 3：护栏 + 审计

1. 在 `canUseTool` 中对危险 Bash/Write 操作加拦截
2. 所有 operations 调用统一写入审计日志
3. UI 端可展示 agent 通过 operations 做了哪些操作

```
产出：
  app/main/services/operations-audit.ts — 审计日志
  app/main/services/agent-service.ts    — canUseTool 配置
```

### 4.4 验证方式

- 单元测试：每个 operation 方法的正常路径 + 异常路径
- 集成测试：通过 IPC 调用 → 验证 service 层状态；通过 MCP 调用 → 验证同样效果
- 手动验证：聊天中说「帮我在 workspace 下建个项目」→ 验证 Mint 使用 `createProject` 工具且被正确拒绝

## 5. 风险与权衡

### 5.1 粒度问题

- **太细**：agent 做一件事需要调用 10 个工具，效率低
- **太粗**：失去灵活性，agent 无法组合操作
- **对策**：从第一批 4 个核心操作开始，实际使用中迭代粒度

### 5.2 Agent 仍然可以绕过

Mint 始终有 Bash/Write 工具——不能完全禁止，否则 agent 无法写代码。Operations Layer 提供的是「更好的路」，不是「封死所有路」。需要 prompt + 护栏配合。

### 5.3 与 SDK 工具的边界

SDK 自带的 `Task` 工具（创建 Subagent）和我们封装的 `createSession` 是不同层级的概念：
- `Task` → SDK 内部 subagent 管理（临时、自动销毁）
- `createSession` → EasyMint 自身的会话记录管理（持久展示在 UI）

### 5.4 渐进式推进

不建议一次性做完所有操作。先做第一批 4 个核心操作，跑通以下链路：

```
Mint 聊天调用 createProject → MCP → Operations Layer → 校验生效 → 项目创建成功
```

验证方案可行后再扩展第二批、第三批。

## 6. 后续讨论要点

- MCP Server 的具体传输方式（stdio / HTTP / in-process）？
- 内建 MCP server 是独立进程还是嵌入主进程？
- Operations 的粒度边界：什么算「基础操作」，什么应该留给 agent 自行组合？
- 是否需要「操作回滚」机制（某些操作失败时撤销已执行步骤）？
- 不同 Agent 角色的 operations 工具分配策略？
