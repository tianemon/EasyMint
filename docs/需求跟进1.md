### ⚠️ 已实现但与原始设计不同（架构升级/替代方案）

| 原始设计 | 当前实现 | 原因 |
|---------|---------|------|
| CLI `spawn("claude")` 子进程 | SDK `query()` + 长生命周期 channel | SDK 提供更好的类型安全、权限控制和生命周期管理 |
| `run-automation.sh` / `evaluate.sh` | Mint 直接调 Task(builder) → Task(evaluator) | Mint 替代 Orchestrator，消除中间层 |
| Worker/Evaluator 为独立 CLI 进程 | SDK Task Subagent（临时创建、自动销毁） | SDK 原生支持，上下文隔离、成本可控 |
| EVALUATOR.md / WORKER.md 操作手册 | Agent 模板 prompt（`agent-templates.ts`） | 模板 prompt 由 SDK 自动注入，无需 Subagent 读文件 |
| Chat 内嵌表单卡片 | NewProjectDialog 弹窗分步表单 | 弹窗方式更简单，等 Chat 卡片方案成熟后再迁移 |
| `progress.txt` 交接日志 | task.json `passes`/`evaluated` 字段 | 结构化字段更可靠，避免文本解析 |
| Orchestrator 调度层 | Mint 直接驱动任务执行 | 减少上下文消耗，简化架构 |

| 流事件内存 buffer | 新窗口挂载时 replay，填补创建间隙 |
| `[系统消息]` 约定 | 自动消息标记，Mint 只执行不聊天 |
| `spawnAgentChat` 独立会话创建 | 从模板创建独立 ActiveChat，注入首条消息 |
| 主进程 `streamBuffer` | session 级别的流事件缓冲，回放后清除 |
| 平台开发规范（`prompts.ts` 内置） | Web/CLI/API 各一套精简规范，初始化时按项目类型注入 |

### 🔮 未来可优化/实现

## 9. GAN 对抗式开发与评估（待设计）

### 当前问题

12 个 UI 任务跑完，Builder 和 Evaluator 形成了一轮对抗后暴露的问题：

- **新旧 UI 混在一起**：多轮迭代导致旧组件残留、CSS 变量多处定义、mock 数据多版本共存
- **小问题堆积**：Evaluator 修了一些但不够彻底，Builder 引入新问题 Evaluator 没发现
- **Evaluator 太客气**：200ms vs 250ms 这种明显差异才 FAIL，更细微的颜色偏差、间距不对齐没抓出来
- **Builder 不够认真**：知道 Evaluator 会兜底，对视觉精度要求放松

### GAN 对抗思路

```
Builder（生成器）                    Evaluator（判别器）
      │                                    │
      ├─ 产出 UI                           ├─ 对照规格逐像素验证
      ├─ 目标：尽可能逼真                  ├─ 目标：找出所有偏差
      │                                    │
      └──────── 相互博弈 ──────────────────┘
              │
              ▼
        Evaluator 越严格 → Builder 越精细
        Builder 越精细 → Evaluator 必须更严格才能发现问题
        循环收敛 → 产出接近像素级还原
```

### 具体措施

1. **Evaluator 强化**：
   - 每个 step 必须产生 2+ 张截图（操作前/后）
   - 颜色偏差 > 5（∆E）即 FAIL，不可放过
   - 组件残留（旧 className、未引用 import）纳入验收
   - 每轮评估报告附带"GAN 分数"：Builder 逼真度 %

2. **Builder 约束**：
   - 开发前先读当前评估报告（知道上次 FAIL 在哪里）
   - 旧代码一并清理（不只是新增，还要删除旧实现）
   - 通过标准从"功能 OK"提升到"视觉规格全匹配"

3. **循环收敛**：
   - 同一任务 FAIL 3 次 → 暂停，人工介入判断规格是否合理
   - PASS 后评估报告归档为回归测试基准

### 明天要做的

- [ ] 分析当前 UI 状态：找出新/旧代码混杂的具体位置
- [ ] 强化 EVALUATOR.md 或创建 GAN_EVALUATOR.md
- [ ] 确定"像素级验收"的可操作标准（色差公式、截图覆盖率）
- [ ] 清理残留旧组件和样式

### 深层问题：不能只靠提示词

开发和评估目前的核心控制手段是 `WORKER.md` 和 `EVALUATOR.md` 两份提示词，外加 task.json 里的 step 描述。这种方式在 12 个任务跑完后暴露出根本性局限：

- **提示词是单向指令**，Builder 和 Evaluator 按照各自的理解执行，缺乏实时反馈闭环
- **评估标准依赖文字描述**，image-vision 返回的是自然语言，不是结构化数据。"颜色是浅绿色"无法和"#f5faf7"做精确对比
- **无参考基准**，Evaluator 不知道"正确的结果长什么样"，只能判断"有没有东西"而非"东西对不对"
- **工具链粗糙**，截图→识图→判断 三个环节各自独立，没有统一的度量框架

### 转型方向：自动化测试体系

这不只是"优化提示词"的问题，需要引入自动化测试领域的成熟方法：

**1. UI 参考图（Baseline Screenshot）**

设计稿或 OD 生成的原型截图作为"标准答案"，Evaluator 拿实际截图做像素级 diff：

```
原型截图（baseline）         实际截图（actual）
      │                          │
      └──── pixel-diff ──────────┘
                │
                ▼
        差异热力图 + 通过阈值 %
```

工具：pixelmatch、resemble.js、looks-same

**2. 结构化视觉对比**

不再依赖 image-vision 的自然语言描述，而是从截图中提取结构化数据：

| 检测项 | 工具 | 输出 |
|--------|------|------|
| 颜色 | 提取指定坐标像素 | hex 值，∆E 色差 |
| 间距 | 边缘检测 + 测距 | px 值 |
| 字体 | OCR + 样式检测 | font-family, size, weight |
| 布局 | DOM 计算样式 | grid/flex 数值 |
| 组件位置 | bounding box 对比 | 偏移量 px |

**3. 循环驱动引擎（替代提示词流程）**

```
┌──────────┐    task.json     ┌──────────┐
│  Builder  │ ──────────────→ │ Evaluator│
│  (生成)   │                 │ (判别)   │
│           │ ←────────────── │          │
│ 修复 FAIL │   评估报告       │ 逐像素对比│
│ 优化实现  │                  │ baseline  │
└──────────┘                  └──────────┘
       │                            │
       └──── harness 控制循环 ──────┘
         · 轮次计数
         · PASS/FAIL 阈值
         · 3 次 FAIL → 人工介入
         · 收敛判定
```

**4. 需要研究的方向**

- pixelmatch 集成：Playwright 截图后跑 pixel-diff，阈值设为 0.5%（即 99.5% 像素一致才算 PASS）
- baseline 管理：每个 task 的 step 对应一张 baseline 截图（来自 OD 原型或设计稿），存在 `temp/evaluator/baselines/task-<id>-<step>.png`
- ∆E 色差计算：Canvas API 取色 + CIE76/CIE94 公式，阈值 < 5
- 回归测试：PASS 后的截图自动归档为下一次的 baseline，防止退化

### 明天要做的（补充）

- [ ] 调研 pixelmatch / resemble.js 集成方案
- [ ] 确定 baseline 截图如何生成（OD 原型截取 vs 手动标注）
- [ ] 设计 harness 控制循环的 Python 代码结构
- [ ] 评估工具链重构：从"截图→自然语言→判断"升级为"截图→结构化对比→数据报告"

## 2. 持久会话架构

### 2.1 核心理念

**每个项目一个持久 Claude 会话。** 从项目创建到功能迭代，始终在同一条 `--resume` 链上。

### 2.2 会话模型

```
项目 A
  Chat 主会话（长会话，--input-format stream-json --output-format stream-json）
    ├── 需求访谈
    ├── 技术选型
    ├── 表单填写
    ├── 启动开发（spawn 子 worker）
    ├── 查看报告
    ├── 新需求讨论
    ├── 再次启动开发
    └── ...永久保持

  Worker 子会话（短会话，-p "按 WORKER.md 完成一个任务"）
    ├── Worker Round 1 → 提交 → 退出
    ├── Worker Round 2 → 提交 → 退出
    └── ...每轮独立，上下文来自文件（CLAUDE.md + task.json + progress.txt）

  Evaluator 子会话（短会话，-p "按 EVALUATOR.md 评估一个任务"）
    └── 每轮 Worker 后可选触发
```

### 2.3 会话生命周期

| 阶段 | 会话 | 说明 |
|------|------|------|
| 项目创建 | spawn 长会话 | `claude --input-format stream-json --output-format stream-json` |
| 需求采集 | 同上 | Chat 面板 = 这个会话的 stdin/stdout |
| 自动化开发 | 同上会话发起 | AI 内部调 `claude -p ...` 异步执行，不阻塞 Chat |
| 暂停/关闭 App | save session | session_id 持久化到项目元数据 |
| 恢复 | `claude --resume <session_id>` | EasyMint 启动时自动 resume |
| 新需求/修改 | 同上 | 用户随时在 Chat 打字，AI 追加 task.json，启动开发 |
| 新建会话讨论 | 可选 | 比如"帮我分析一下性能问题"，同样自动注入项目上下文 |

### 2.4 上下文管理

每个项目维护一个上下文快照文件 `~/.easymint/<project-id>/context.json`：

```json
{
  "project_id": "uuid",
  "claude_session_id": "sess-abc123",
  "created_at": "2026-05-28T10:00:00Z",
  "last_active_at": "2026-05-28T14:30:00Z",
  "profile": {
    "name": "个人记账 App",
    "type": "mobile",
    "platform": ["ios", "android"],
    "tech_stack": {
      "framework": "react-native",
      "styling": "tailwind",
      "backend": "firebase"
    },
    "requirements_summary": "iOS+Android 记账应用，支持分类统计、预算提醒、云同步、多设备",
    "completed_tasks": 15,
    "total_tasks": 18
  },
  "recent_conversations": [
    {
      "id": "conv-3",
      "title": "添加预算提醒功能",
      "summary": "用户要求添加月度预算设置和超支提醒，新增 3 个 task",
```

**注入时机**：

- 新 Chat 会话创建时 → context.json 作为 system prompt 注入
- Worker 子会话启动时 → 不需要注入（它读 CLAUDE.md + WORKER.md + task.json）
- 项目恢复时 → `claude --resume` 自动恢复完整上下文

**更新时机**：

- 需求采集完成 → 更新 profile
- 任务完成 → 更新 completed_tasks，摘录 progress.txt 关键行
- 新对话结束 → 追加 recent_conversations，AI 自动生成摘要
- 文件变更 → 更新 active_file_list

## 3. 对话驱动表单系统

### 3.1 核心思路

**Chat 是主线，表单是辅助。** AI 通过对话理解用户意图，根据对话进程按需调用对应的结构化表单卡片。用户可以在卡片上直接操作，也可以继续打字让 AI 代填。

不是"用户填完表单 → AI 看结果"，而是"AI 和用户一起填"。

### 3.2 表单流转示例

```

  │ 类型: 移动端 App             │

用户修改名称 → 填描述 → 点确认
  → 表单数据 JSONL → AI

AI 消息："明白了。移动端我建议用 React Native 同时覆盖两端，Firebase 做后
  端可以省掉服务器成本。你怎么看？" + 附带技术选型卡片

  ┌─────────────────────────────┐
  │ 🔧 技术选型                  │
  │                              │
  │ 前端框架:                     │
  │ ● React Native ← AI 推荐     │
```

### 3.3 表单模板库

预置一组可组合的 form schema，每个 schema 是声明式 JSON，定义字段、校验规则、AI 推荐逻辑。不是写死的表单代码。

```
form-templates/
├── overview           # 项目概述：名称、平台选择、类型、描述
├── tech-frontend      # 前端框架选择（按项目类型提供不同选项）
│   ├── web            # React / Vue / Next.js / Nuxt / Svelte / 纯 HTML
│   ├── mobile         # React Native / Flutter / Swift+Kotlin / Expo
│   ├── desktop        # Electron / Tauri / WPF / SwiftUI
│   └── cli            # Node.js / Python / Go / Rust
├── tech-backend       # 后端方案：Firebase / Supabase / 自建API / 无后端
├── tech-styling       # 样式：Tailwind / CSS Modules / Styled Components / 原生
├── database           # 数据库：PostgreSQL / MySQL / SQLite / MongoDB / 无
├── auth               # 认证：Firebase Auth / NextAuth / 自建 / 不需要
├── features           # 功能清单 P0/P1/P2（AI 根据项目类型预推荐）
├── ui-style           # UI 风格：色彩倾向、排版、组件库偏好
├── deploy-web         # Web 部署：Vercel / Netlify / Cloudflare / 自有服务器
├── deploy-mobile      # 移动端发布：App Store / Google Play / TestFlight / 企业分发
├── deploy-desktop     # 桌面端分发：安装包 / 应用商店 / 压缩包
├── budget             # 技术费用：充足 / 少量 / 完全免费
└── confirmation       # 汇总确认：展示所有选择的 summary
```

### 3.4 Form Schema 设计

每个 schema 包含：

```json
{
  "id": "tech-frontend-mobile",
  "title": "选择前端框架",
  "description": "用于构建 iOS 和 Android 应用",
  "trigger_keywords": ["移动端", "App", "iOS", "Android", "Flutter", "React Native"],
  "depends_on": ["overview"],
  "condition": "project.type === 'mobile'",
  "fields": [
    {
      "key": "framework",
      "label": "前端框架",
      "type": "radio",
      "ai_recommend": "react-native",
      "options": [
        {
          "value": "react-native",
          "label": "React Native",
          "description": "基于 React，社区活跃，支持 Expo 快速原型",
          "cost_note": "免费开源"
        },
        {
          "value": "flutter",
          "label": "Flutter",
          "description": "Google 出品，性能接近原生，Dart 语言",
          "cost_note": "免费开源"
        },
        {
          "value": "native",
          "label": "Swift + Kotlin（原生）",
          "description": "最高性能，但需维护两套代码",
          "cost_note": "开发成本较高"
        }
      ]
    }
  ]
}
```

AI 根据对话进程决定：
- 当前进度对应的 schema
- 该 schema 是否需要弹出（condition 字段）
- 哪些字段可以 AI 预填（ai_recommend 字段）
- 弹出时机：一次性弹一个还是多个卡片并列

### 3.5 自适应规则

| 项目类型 | 需要的表单 | 跳过的表单 |
|---------|-----------|-----------|
| Web 前端 | overview + tech-frontend(web) + tech-backend + tech-styling + features + ui-style + deploy-web + budget + confirmation | deploy-mobile, deploy-desktop |
| 移动端 App | overview + tech-frontend(mobile) + tech-backend + tech-styling + features + ui-style + deploy-mobile + budget + confirmation | deploy-web, deploy-desktop |
| CLI / API | overview + tech-frontend(cli) + tech-backend + database + features + budget + confirmation | ui-style, deploy-*, tech-styling |
| 桌面应用 | overview + tech-frontend(desktop) + tech-backend + tech-styling + features + ui-style + deploy-desktop + budget + confirmation | deploy-mobile, deploy-web |
| 全栈（Web + 后端） | 全部，但如果 AI 推荐同一供应商则合并提示 | — |

### 3.6 用户自由对话的容错

用户不一定要在卡片上操作。如果用户放弃表单、直接打字：

```
用户："算了我不想选了，你帮我全定了吧，我只要便宜就行"

AI：
- budget → "完全免费"
- framework → 按类型推荐免费方案
- backend → 选免费额度最高的
- 在 confirmation 卡片上展示全 AI 选择，用户确认即可
```

### 3.7 Chat 消息渲染 — 参考 open-design

Chat 面板的消息渲染采用 open-design 的分层展示模式：

**外层：最终对话流**
用户看到的是一条干净的双向对话。AI 的每条最终回复（text/markdown）直接展示，与用户消息交替排列，像聊天一样自然。

**内层：折叠的过程细节**
每条 AI 消息背后可能包含多个步骤，每个步骤默认折叠，点击标题展开：

- **思考过程** — AI 的 reasoning/thinking 内容，折叠为一行灰色小字 "🤔 思考中..."
- **工具调用** — Read/Write/Edit/Bash 等操作，折叠为 "🔧 Read file_path=xxx" 或 "📝 Edit app/xxx.tsx"
- **文件变更** — diff 摘要，折叠为 "📄 修改了 3 个文件"
- **子任务启动** — 启动 Worker/Evaluator，折叠为 "🚀 启动开发 — Task #5"

展开后可以看到完整的思考文本、工具参数/结果、diff 详情。

**视觉层次**：
```
🤖 AI: 好的，我来帮你实现这个功能。

  [展开] 🤔 思考过程
        分析了需求，这需要一个表单组件，先看看现有代码结构...

  [展开] 🔧 Read — app/src/App.tsx
```

## 5. 实现计划

### 阶段 1 — UI 布局（当前）

纯 UI + Mock 数据，不接入 Claude。确认交互流畅、设计一致。

### 阶段 2 — 基础 Chat

实现真正的 Chat 面板 + 长会话 spawn。Mock 数据替换为实际 JSONL 流。

### 阶段 3 — 表单系统

表单模板库 + 对话内嵌卡片 + AI 路由逻辑。先做 Web 类型的表单链，再扩展移动端/CLI。

### 阶段 4 — 自动化流水线

agent-service 完整实现：worker 调度、evaluator 触发、progress 流转。同时把 `run-automation.sh` 和 `evaluate.sh` 用 Python 重写。

#### 为什么改用 Python

当前 bash 脚本的问题：

- **JSON 操作靠内联 python3 -c**，每次读 task.json 都要 spawn 一个 Python 子进程
- **轮询/超时/grace period 逻辑** 在 bash 里用 while + sleep + 时间戳比较，可读性和可维护性差
- **错误处理散落 `|| true`**，容易掩盖真实问题
- **跨平台**：bash 在 Windows 上需 WSL，Python 直接跑

#### 重写后结构

```
run_automation.py <runs> [--evaluate]   # 替代 run-automation.sh
evaluate.py [--task <id>]               # 替代 evaluate.sh
init.sh                                  # 保持不变，就几行环境检测
```

#### Python 脚本职责

`run_automation.py`：
- 读取 task.json → 选未完成任务
- spawn claude 子进程（带 sandbox 包裹）
- 轮询检查 task.json passes 字段
- 检测到 passes:true → 等待 Claude 自然退出（grace period）
- --evaluate 模式下每轮后触发 evaluator
- 日志写结构化 JSON，进度同步到 progress.json

`evaluate.py`：
- 读取 task.json → 选待评估任务
- spawn claude 子进程（按 EVALUATOR.md 评估）
- 等待完成后读取报告文件
- 更新 task.json evaluated 字段

#### 关键接口

```python
# 读取任务状态 — 不再 python3 -c 内联
with open("task.json") as f:
    tasks = json.load(f)["tasks"]

# 轮询 — Python 原生控制流
task_done_at = None
while True:
    time.sleep(60)
    if check_task_done(task_id):
        task_done_at = time.time()
        # 等 Claude 自然退出...
    if task_done_at and (time.time() - task_done_at) > 120:
        process.kill()  # 超时强制终止

# 错误处理 — 明确的 try/except
try:
    process = subprocess.Popen(...)
except FileNotFoundError:
    log.error("Claude CLI 未找到")
```

#### 保留的文件

- `init.sh` — 保持 bash，功能简单且需要用户终端直接执行
- `run-automation.sh` / `evaluate.sh` — 可保留为对 Python 的轻量 wrapper（兼容原有习惯）
- 跨平台（Windows 也能跑），`init.sh` 保持 bash

### 阶段 5 — 上下文管理

context.json 维护、会话 resume、新会话自动注入。

### 阶段 6 — 持续性开发

task.json 动态追加、增量开发、历史会话管理。

### 阶段 7 — CLI → SDK 迁移

当前 `agent-service.ts` 通过 `child_process.spawn("claude")` 运行 Worker/Evaluator。迁移至 `@anthropic-ai/claude-agent-sdk` 的 `sdk.query()`。

#### 动机

| | CLI spawn（现状） | SDK `query()` |
|---|---|---|
| 进程管理 | 手动 spawn/kill/wait/grace period | SDK 内部管理 |
| 输出格式 | 解析 JSONL 字符串 | 类型安全 `SDKMessage` |
| 权限控制 | `bypassPermissions` 一刀切 | `allowedTools` 白名单 + `canUseTool` 逐项决策 + `hooks` 拦截 |
| 沙箱 | 外部 `sandbox-exec` 包裹 | 内建 `sandbox` 配置 |
| 会话 resume | 手动拼命令 | `query({ resume: sessionId })` |
| 生命周期 | 进程级 | Hook 回调（PreToolUse/PostToolUse/Stop/SessionEnd） |
| 依赖 | 用户安装 `claude` CLI | npm 包，零用户配置 |
| 审计 | 自己写 | Hook 天然支持 |
| 费用 | 消耗 Pro/Max 订阅额度 | 配 DeepSeek API Key 走 DeepSeek 账单，无 Anthropic 费用 |

#### 改动范围

```
agent-service.ts      → 删掉 spawn/JSONL 解析，改为 sdk.query() for-await 循环
jsonl-parser.ts       → 可移除（SDK 已返回结构化消息）
run-automation.sh/py  → 简化，不再管进程超时/kill/grace period
claude-detector.ts    → 改为检测模块：SDK 可用 + 可选 CLI fallback
```

#### 兼容性

- DeepSeek 通过 `ANTHROPIC_BASE_URL` + `ANTHROPIC_API_KEY` 环境变量接入，SDK 和 CLI 走同一套底层
- 安装为 npm 依赖，无需用户手动装 CLI
- 权限细粒度：允许 Read/Write/Edit，禁止 Bash `rm -rf`/`sudo`/`curl | bash`，Hook 实时拦截

### 阶段 8 — EM + OD 协作

EasyMint 接入 open-design，需求确认后、开发开始前，可选生成 UI 原型。原型确认后再启动自动化开发，确保首轮结果符合用户预期。

## 7. 流程控制与稳定性

### 7.1 行为监控

所有 Worker 和 Evaluator 的运行状态必须实时可观测：

- **Worker 监控**：当前任务 ID、开始时间、已耗时、JSONL 事件计数、最后事件类型
- **Evaluator 监控**：当前评估的任务、截图数、image-vision 调用数、当前状态
- **进程健康**：子进程 PID、CPU/内存占用、是否僵死
- **监控面板**：StreamPanel 上方或侧边显示简版状态栏，异常时变色
- **事件日志**：所有 spawn/exit/error/kill 事件记入结构化日志

### 7.2 开发锁 + 需求队列

开发进行中锁定项目，新需求排队，本轮完成后统一处理：

```
用户提需求
  → 如果开发进行中：
    → 需求进入队列（显示 "需求已加入队列 #3"）
    → 队列 UI：右侧面板或 Chat 侧边，可拖拽排序
  → 如果开发未运行：
    → 直接追加 task.json → 启动新一轮

本轮开发完成
  → 自动 flush 队列：
    → 按优先级顺序将积攒需求合并到 task.json
    → 启动新一轮开发
    → 队列清空
```

- **队列可管理**：用户可调整优先级、删除、合并需求
- **队列可见**：开发中显示 "当前开发中 + 3 个需求排队"
- **强制模式**：用户可手动暂停开发、清空队列、或立即停止当前任务插入紧急需求

### 7.3 终止流程完善

用户点「停止」后必须保证：

1. SIGTERM → 等待 5 秒 → SIGKILL（给子进程清理时间）
2. 清理子进程树（包括 spawn 的孙子进程）
3. 保存当前进度（task.json 不丢，passes 状态不丢）
4. StreamPanel 显示 "已停止" + 当前进度摘要
5. 释放文件锁
6. Chat 会话不受影响（Worker 死了 Chat 还在）

### 7.4 并发写保护

task.json 是唯一真相源，任何时刻只能一个进程写入：

- **文件锁**：写入前 `flock` / 创建 `.lock` 文件
- **原子写**：先写 `.tmp` → `fs.rename` → 删除旧文件
- **冲突检测**：读取时检查 lock，等待或报错
- **优先级**：用户手动操作 > Worker 自动操作 > Evaluator

### 7.5 进度日志结构化

`progress.txt` 替换为 `progress.json`：

## 8. 沙箱隔离

### 8.1 问题

Worker 和 Evaluator 需要 `--permission-mode bypassPermissions` 才能自动化运行，但 bypass 之后 CLI 内建的权限弹窗不会触发，CLAUDE.md 的安全约束本质上靠 prompt 遵守，没有物理兜底。

### 8.2 方案

**macOS：sandbox-exec + 动态 .sb 配置**

```bash
# EasyMint 在 spawn 前根据 project.path 动态生成 sandbox 配置
cat > /tmp/easymint-worker-${runId}.sb <<EOF
(version 1)
(deny default)
(allow file-read* file-write* (subpath "${project.path}"))
(allow file-read* (subpath "/opt/homebrew"))
(allow file-read* (subpath "/usr/local"))
(allow file-read* (subpath "/private/var/db/xcode_select_link"))
(allow process*)
(allow sysctl*)
(allow network* (local ip))
EOF

sandbox-exec -f /tmp/easymint-worker-${runId}.sb \
  claude -p "..." --permission-mode bypassPermissions
```

**Windows：AppContainer + Low Integrity Level**

```powershell
# 首次创建隔离容器
New-AppContainerProfile -Name "EasyMintWorker" -DisplayName "EasyMint Worker" ...

# spawn 时限定
Start-Process -AppContainer "EasyMintWorker" -Wait -NoNewWindow claude ...
```

### 8.3 能力白名单

| 操作 | 是否允许 | 原因 |
|------|---------|------|
| 读写目标项目目录 | ✅ | worker 只需操作当前项目 |
| 读 node/npm/claude 工具链 | ✅ | 开发工具链路径 |
| npm install（外网） | ✅ | 依赖下载 |
| Vite dev server localhost | ✅ | 开发预览 |
| curl API 测试 | ✅ | 后端验证 |
| git commit | ✅ | 在项目目录内的 .git 操作 |
| 读 ~/.ssh 私钥 | ❌ | 禁止访问用户凭证 |
| 读 ~/Documents、~/Desktop | ❌ | 禁止访问用户数据 |
| 写 /etc、/System | ❌ | 禁止修改系统 |
| 读写其他项目目录 | ❌ | 项目间物理隔离 |
| 外网 curl（非必要） | ❌ | 默认禁止，按需放开 |

### 8.4 关键设计决策

- **项目不动**：沙箱管进程，不管文件。项目仍在用户选的目录，Worker 被物理锁在该目录内
- **动态配置**：每个 Worker/Evaluator spawn 时根据 project.path 动态生成临时沙箱配置
- **平台适配**：macOS sandbox-exec（内建），Windows AppContainer（内建），跨平台无需额外依赖
- **最坏情况**：bypassPermissions 模式下 Claude 即便完全失控，也只能毁掉当前项目，碰不到用户的其他数据
