# 需求跟进 #2 — AI 驱动的完整开发平台

> 2026-05-28 — EasyMint 从"模板 GUI 壳"升级为"AI 驱动的完整开发平台"

---

## 1. 愿景

EasyMint 不再只是模板项目的 GUI 外壳，而是一个完整的 AI 开发平台。用户打开 App → 对话式创建项目 → 自动化开发 → 评估报告 → 持续性迭代，整个过程在一个持久会话中完成。

模板（ai-coding-automation-template）从"用户可见的产物"变为"EasyMint 内部的工作协议"——用户不知道也不关心模板存在，他只跟 AI 对话。

### 用户视角的完整流程

```
打开 EasyMint
  → Onboarding（检测 Claude Code，仅首次）
  → 进入工作台

  → 新建项目 / 打开已有项目

  → 【Chat 面板】与 AI 对话：
     "我想做一个 iOS 和 Android 都能用的记账 App"

  → AI 分析需求，逐步引导：
     问关键问题 → 内部调用对应表单 → 表单数据回传 AI → 继续对话

  → 需求确认完毕，AI 生成 requirements.md
     （内部执行：脚手架搭建 → task.json 生成 → init.sh 填充）

  → 用户点击「启动开发」
     → spawn 子会话循环跑 task.json
     → 每轮 builder + evaluator → 报告实时展示在 StreamPanel

  → 项目"完成"只是阶段性的：
     → 用户继续在 Chat 面板提新需求
     → AI 追加/修改 task.json
     → 启动新一轮自动化开发

  → 任何时候，新建 Chat 会话
     → AI 自动加载项目上下文（技术栈、已完成任务、会话摘要）
     → 像一位一直跟着项目的工程师一样接上
```

## 2. 持久会话架构

### 2.1 核心理念

**每个项目一个持久 Claude 会话。** 从项目创建到功能迭代，始终在同一条 `--resume` 链上。

### 2.2 会话模型

```
项目 A
  Chat 主会话（长会话，--input-format stream-json --output-format stream-json）
    ├── 需求访谈
    ├── 技术选型
    ├── 表单填写
    ├── 启动开发（spawn 子 worker）
    ├── 查看报告
    ├── 新需求讨论
    ├── 再次启动开发
    └── ...永久保持

  Worker 子会话（短会话，-p "按 WORKER.md 完成一个任务"）
    ├── Worker Round 1 → 提交 → 退出
    ├── Worker Round 2 → 提交 → 退出
    └── ...每轮独立，上下文来自文件（CLAUDE.md + task.json + progress.txt）

  Evaluator 子会话（短会话，-p "按 EVALUATOR.md 评估一个任务"）
    └── 每轮 Worker 后可选触发
```

### 2.3 会话生命周期

| 阶段 | 会话 | 说明 |
|------|------|------|
| 项目创建 | spawn 长会话 | `claude --input-format stream-json --output-format stream-json` |
| 需求采集 | 同上 | Chat 面板 = 这个会话的 stdin/stdout |
| 自动化开发 | 同上会话发起 | AI 内部调 `claude -p ...` 异步执行，不阻塞 Chat |
| 暂停/关闭 App | save session | session_id 持久化到项目元数据 |
| 恢复 | `claude --resume <session_id>` | EasyMint 启动时自动 resume |
| 新需求/修改 | 同上 | 用户随时在 Chat 打字，AI 追加 task.json，启动开发 |
| 新建会话讨论 | 可选 | 比如"帮我分析一下性能问题"，同样自动注入项目上下文 |

### 2.4 上下文管理

每个项目维护一个上下文快照文件 `~/.easymint/<project-id>/context.json`：

```json
{
  "project_id": "uuid",
  "claude_session_id": "sess-abc123",
  "created_at": "2026-05-28T10:00:00Z",
  "last_active_at": "2026-05-28T14:30:00Z",
  "profile": {
    "name": "个人记账 App",
    "type": "mobile",
    "platform": ["ios", "android"],
    "tech_stack": {
      "framework": "react-native",
      "styling": "tailwind",
      "backend": "firebase"
    },
    "requirements_summary": "iOS+Android 记账应用，支持分类统计、预算提醒、云同步、多设备",
    "completed_tasks": 15,
    "total_tasks": 18
  },
  "recent_conversations": [
    {
      "id": "conv-3",
      "title": "添加预算提醒功能",
      "summary": "用户要求添加月度预算设置和超支提醒，新增 3 个 task",
      "created_at": "2026-05-28T14:00:00Z"
    }
  ],
  "active_file_list": [
    "src/components/BudgetChart.tsx",
    "src/screens/AddTransaction.tsx"
  ]
}
```

**注入时机**：

- 新 Chat 会话创建时 → context.json 作为 system prompt 注入
- Worker 子会话启动时 → 不需要注入（它读 CLAUDE.md + WORKER.md + task.json）
- 项目恢复时 → `claude --resume` 自动恢复完整上下文

**更新时机**：

- 需求采集完成 → 更新 profile
- 任务完成 → 更新 completed_tasks，摘录 progress.txt 关键行
- 新对话结束 → 追加 recent_conversations，AI 自动生成摘要
- 文件变更 → 更新 active_file_list

## 3. 对话驱动表单系统

### 3.1 核心思路

**Chat 是主线，表单是辅助。** AI 通过对话理解用户意图，根据对话进程按需调用对应的结构化表单卡片。用户可以在卡片上直接操作，也可以继续打字让 AI 代填。

不是"用户填完表单 → AI 看结果"，而是"AI 和用户一起填"。

### 3.2 表单流转示例

```
用户："我想做一个记账 App，iOS 和 Android 都要"

AI 分析 → 提取特征：
  类型=移动端, 平台=iOS+Android, 领域=财务

AI 消息："好的！我先帮你梳理一下基本需求。" + 附带项目概述卡片
  ┌─────────────────────────────┐
  │ 📋 项目概述                  │
  │ 名称: [记账 App_____]        │
  │ 平台: ☑ iOS  ☑ Android     │
  │ 类型: 移动端 App             │
  │ 描述: [___________________] │
  └─────────────────────────────┘

用户修改名称 → 填描述 → 点确认
  → 表单数据 JSONL → AI

AI 消息："明白了。移动端我建议用 React Native 同时覆盖两端，Firebase 做后
  端可以省掉服务器成本。你怎么看？" + 附带技术选型卡片

  ┌─────────────────────────────┐
  │ 🔧 技术选型                  │
  │                              │
  │ 前端框架:                     │
  │ ● React Native ← AI 推荐     │
  │ ○ Flutter                    │
  │ ○ Swift + Kotlin（原生）      │
  │                              │
  │ 后端方案:                     │
  │ ● Firebase（免费额度够用）     │
  │ ○ Supabase                   │
  │ ○ 自建 API + 数据库           │
  │                              │
  │ 样式方案:                     │
  │ ● Tailwind (NativeWind)      │
  │ ○ StyleSheet                 │
  └─────────────────────────────┘

用户打字："用 Flutter 吧，我朋友说性能更好"

AI 自动切换选中 Flutter → 更新卡片 → 继续追问："Flutter 的话后端还是 Firebase？"

...
```

### 3.3 表单模板库

预置一组可组合的 form schema，每个 schema 是声明式 JSON，定义字段、校验规则、AI 推荐逻辑。不是写死的表单代码。

```
form-templates/
├── overview           # 项目概述：名称、平台选择、类型、描述
├── tech-frontend      # 前端框架选择（按项目类型提供不同选项）
│   ├── web            # React / Vue / Next.js / Nuxt / Svelte / 纯 HTML
│   ├── mobile         # React Native / Flutter / Swift+Kotlin / Expo
│   ├── desktop        # Electron / Tauri / WPF / SwiftUI
│   └── cli            # Node.js / Python / Go / Rust
├── tech-backend       # 后端方案：Firebase / Supabase / 自建API / 无后端
├── tech-styling       # 样式：Tailwind / CSS Modules / Styled Components / 原生
├── database           # 数据库：PostgreSQL / MySQL / SQLite / MongoDB / 无
├── auth               # 认证：Firebase Auth / NextAuth / 自建 / 不需要
├── features           # 功能清单 P0/P1/P2（AI 根据项目类型预推荐）
├── ui-style           # UI 风格：色彩倾向、排版、组件库偏好
├── deploy-web         # Web 部署：Vercel / Netlify / Cloudflare / 自有服务器
├── deploy-mobile      # 移动端发布：App Store / Google Play / TestFlight / 企业分发
├── deploy-desktop     # 桌面端分发：安装包 / 应用商店 / 压缩包
├── budget             # 技术费用：充足 / 少量 / 完全免费
└── confirmation       # 汇总确认：展示所有选择的 summary
```

### 3.4 Form Schema 设计

每个 schema 包含：

```json
{
  "id": "tech-frontend-mobile",
  "title": "选择前端框架",
  "description": "用于构建 iOS 和 Android 应用",
  "trigger_keywords": ["移动端", "App", "iOS", "Android", "Flutter", "React Native"],
  "depends_on": ["overview"],
  "condition": "project.type === 'mobile'",
  "fields": [
    {
      "key": "framework",
      "label": "前端框架",
      "type": "radio",
      "ai_recommend": "react-native",
      "options": [
        {
          "value": "react-native",
          "label": "React Native",
          "description": "基于 React，社区活跃，支持 Expo 快速原型",
          "cost_note": "免费开源"
        },
        {
          "value": "flutter",
          "label": "Flutter",
          "description": "Google 出品，性能接近原生，Dart 语言",
          "cost_note": "免费开源"
        },
        {
          "value": "native",
          "label": "Swift + Kotlin（原生）",
          "description": "最高性能，但需维护两套代码",
          "cost_note": "开发成本较高"
        }
      ]
    }
  ]
}
```

AI 根据对话进程决定：
- 当前进度对应的 schema
- 该 schema 是否需要弹出（condition 字段）
- 哪些字段可以 AI 预填（ai_recommend 字段）
- 弹出时机：一次性弹一个还是多个卡片并列

### 3.5 自适应规则

| 项目类型 | 需要的表单 | 跳过的表单 |
|---------|-----------|-----------|
| Web 前端 | overview + tech-frontend(web) + tech-backend + tech-styling + features + ui-style + deploy-web + budget + confirmation | deploy-mobile, deploy-desktop |
| 移动端 App | overview + tech-frontend(mobile) + tech-backend + tech-styling + features + ui-style + deploy-mobile + budget + confirmation | deploy-web, deploy-desktop |
| CLI / API | overview + tech-frontend(cli) + tech-backend + database + features + budget + confirmation | ui-style, deploy-*, tech-styling |
| 桌面应用 | overview + tech-frontend(desktop) + tech-backend + tech-styling + features + ui-style + deploy-desktop + budget + confirmation | deploy-mobile, deploy-web |
| 全栈（Web + 后端） | 全部，但如果 AI 推荐同一供应商则合并提示 | — |

### 3.6 用户自由对话的容错

用户不一定要在卡片上操作。如果用户放弃表单、直接打字：

```
用户："算了我不想选了，你帮我全定了吧，我只要便宜就行"

AI：
- budget → "完全免费"
- framework → 按类型推荐免费方案
- backend → 选免费额度最高的
- 在 confirmation 卡片上展示全 AI 选择，用户确认即可
```

### 3.7 Chat 消息渲染 — 参考 open-design

Chat 面板的消息渲染采用 open-design 的分层展示模式：

**外层：最终对话流**
用户看到的是一条干净的双向对话。AI 的每条最终回复（text/markdown）直接展示，与用户消息交替排列，像聊天一样自然。

**内层：折叠的过程细节**
每条 AI 消息背后可能包含多个步骤，每个步骤默认折叠，点击标题展开：

- **思考过程** — AI 的 reasoning/thinking 内容，折叠为一行灰色小字 "🤔 思考中..."
- **工具调用** — Read/Write/Edit/Bash 等操作，折叠为 "🔧 Read file_path=xxx" 或 "📝 Edit app/xxx.tsx"
- **文件变更** — diff 摘要，折叠为 "📄 修改了 3 个文件"
- **子任务启动** — 启动 Worker/Evaluator，折叠为 "🚀 启动开发 — Task #5"

展开后可以看到完整的思考文本、工具参数/结果、diff 详情。

**视觉层次**：
```
🤖 AI: 好的，我来帮你实现这个功能。

  [展开] 🤔 思考过程
        分析了需求，这需要一个表单组件，先看看现有代码结构...

  [展开] 🔧 Read — app/src/App.tsx
         读取主入口文件，确认路由结构...

  [展开] 📝 Edit — app/src/components/NewForm.tsx
         创建新组件，包含表单逻辑...

  AI 最终回复: 已完成。新建了 NewForm 组件，路由已接入，你可以在 /new 看到。

👤 用户: 能加上表单验证吗？

🤖 AI: 当然可以。

  [展开] 🤔 思考过程
        需要添加验证逻辑...

  [展开] 📝 Edit — app/src/components/NewForm.tsx
         添加 required + pattern 验证...

  AI 最终回复: 已添加验证，包括必填字段和邮箱格式校验。
```

**关键设计决策**：
- 主对话流始终可见，不被过程细节打断
- 过程默认折叠，用户想看细节才点开
- 思考/操作/编辑三个层级用不同颜色/图标区分
- 过程步骤按时间顺序排列，不交错

### 3.8 表单卡片 UI

表单卡片内嵌在 Chat 消息流中，是 AI 消息的一种特殊渲染形态：

```
Chat 消息流：
┌──────────────────────────────────┐
│ 🤖 AI: 移动端我建议这几个方案...    │
```

## 4. 自动化开发流水线

### 4.1 流程

SETUP Phase 4-5 的逻辑内化为 EasyMint 的调度器，模板的 `run-automation.sh` 不再由用户执行，而是由 EasyMint 内部调度：

```
用户点击「启动开发」
  → agent-service 读取 task.json
  → 选一个 passes: false 的任务
  → spawn claude -p "按 WORKER.md 完成一个任务" --output-format stream-json
  → stdout 通过 StreamPanel 实时展示
  → 任务完成（passes → true）
  → 如果评估模式开启 → spawn evaluator
  → 下一轮...
  → 直到全部 passes: true
```

### 4.2 进度可视化

- **StreamPanel**：实时展示 builder/evaluator 的 JSONL 流（文本、工具调用、文件变更）
- **右面板任务列表**：任务状态（pending / running / done / failed），点击查看步骤详情
- **Chat 面板**：AI 在每轮完成后发消息总结："已完成 Task 3/5，开始 Task 4..."

### 4.3 持续性开发

项目"完成"（全部 passes: true）不是终点：

```
用户: "这个记账 App 能加上语音记账吗？"

AI:
  1. 分析需求
  2. 生成新 task(s) — 追加到 task.json
  3. 回复: "好的，语音记账需要 3 个新任务：语音识别接入、数据解析、UI 适配。
            要现在启动开发吗？"

用户: "开始"

  → 新一轮自动化开发（只做新追加的任务）
```

## 5. 实现计划

### 阶段 1 — UI 布局（当前）

纯 UI + Mock 数据，不接入 Claude。确认交互流畅、设计一致。

### 阶段 2 — 基础 Chat

实现真正的 Chat 面板 + 长会话 spawn。Mock 数据替换为实际 JSONL 流。

### 阶段 3 — 表单系统

表单模板库 + 对话内嵌卡片 + AI 路由逻辑。先做 Web 类型的表单链，再扩展移动端/CLI。

### 阶段 4 — 自动化流水线

agent-service 完整实现：worker 调度、evaluator 触发、progress 流转。

同步用 Python 重写 `run-automation.sh` 和 `evaluate.sh`：
- JSON 操作原生支持，不再 `python3 -c` 内联
- 轮询/超时/grace period 逻辑用 Python 控制流
- 错误处理 try/except，减少 `|| true` 容错
- 跨平台（Windows 也能跑），`init.sh` 保持 bash

### 阶段 5 — 上下文管理

context.json 维护、会话 resume、新会话自动注入。

### 阶段 6 — 持续性开发

task.json 动态追加、增量开发、历史会话管理。

### 阶段 7 — EM + OD 协作

EasyMint 接入 open-design，需求确认后、开发开始前，可选生成 UI 原型。原型确认后再启动自动化开发，确保首轮结果符合用户预期。

## 6. EM + OD 协作：开发前先看到效果

### 6.1 核心思路

用户描述想法 → EM 采集需求 → **可选**：OD 生成 UI 原型 → 用户调整 UI → OD 反哺 EM 更新规格 → EM 启动开发 → evaluator 拿原型做验收参照。

不是"开发完再看对不对"，而是"看清了再开始开发"。

### 6.2 完整流程

```
EM: Chat 对话 + 表单采集
  → 生成 requirements.md
                    │
      ┌─────────────┼─────────────┐
      ▼                           ▼
  需要 UI                      不需要 UI
  (Web / 移动 / 桌面)          (CLI / API / 服务端)
      │                           │
  EM 弹出选项：                    │
  ┌──────────────────────┐        │
  │ 🎨 先去 Open Design  │        │
  │ 生成 UI 原型，确认后再│        │
  │ 启动开发              │        │
  │                      │        │
  │ [  生成 UI 原型  ]   │  ← 主按钮，高亮
  │ [ 直接开始开发    ]   │  ← 次要
  └──────────────────────┘        │
      │                           │
  选"生成 UI"                     │
      ▼                           │
  EM 调用 OD：                     │
    将 requirements.md + 技术栈    │
    + UI 偏好作为提示词            │
      ▼                           │
  OD 生成可交互 HTML 原型           │
    → 用户在 OD 中预览              │
    → 用户调整布局、颜色、组件       │
    → 满意后确认                    │
      ▼                           │
  OD 反哺 EM：                     │
    · 原型截图 → evaluator 参照图   │
    · 视觉规格（颜色/间距/组件清单）  │
    → 更新 APP_SPEC               │
    → 更新 task.json               │
      （每个 step 带具体视觉验收条件） │
      │                           │
      └─────────────┬─────────────┘
                    ▼
            启动自动化开发
    （task.json 已包含视觉标准，
     evaluator 有原型截图做参照）
```

### 6.3 判断逻辑

| 项目类型 | EM 默认行为 |
|---------|-----------|
| Web 前端 | 强烈建议 → "生成 UI 原型" 按钮高亮，推荐文案 |
| 移动端 App | 强烈建议 → 同上 |
| 桌面应用 | 建议 → 两个按钮并列 |
| CLI / API / 服务端 | 不弹选项 → 直接显示"下一步" |

EM 内部判断规则：

```
if project.type in ['web', 'mobile', 'desktop']:
    if project.type == 'desktop':
        show_ui_option(prominence='normal')  # 两个按钮并列
    else:
        show_ui_option(prominence='strong')  # 主按钮高亮
else:
    skip_ui_option()  # 直接下一步
```

### 6.4 数据流

```
EM → OD:
  {
    "requirements": "<requirements.md 内容>",
    "tech_stack": { "framework": "react", "styling": "tailwind" },
    "features": ["P0: xxx", "P1: yyy"],
    "ui_preferences": { "color": "极简白", "typography": "无衬线" }
  }

OD → EM（用户确认后）:
  {
    "prototype_url": "/path/to/prototype.html",
    "screenshots": ["homepage.png", "detail.png", "modal.png"],
    "visual_spec": {
      "colors": { "bg": "#ffffff", "primary": "#2563eb", ... },
      "spacing": { "card_gap": "16px", "section_padding": "24px" },
      "components": ["Header", "PostCard", "Layout", ...]
    }
  }
```

### 6.5 反哺规则

OD 返回的 visual_spec 自动映射到 task.json 的验收条件：

| visual_spec | 映射到 |
|------------|--------|
| colors.bg = #ffffff | "页面背景为纯白 #ffffff" |
| spacing.card_gap = 16px | "卡片间距为 16px" |
| components: [Header, PostCard] | "Header 组件存在且渲染正确"、"PostCard 组件存在且渲染正确" |
| screenshots | evaluator 参照图路径 |

### 6.6 用户体验要点

- **无缝切换**：EM 打开 OD 时保持上下文，用户在 OD 调完 UI 回来，EM 的 Chat 画面还在
- **不强制**：纯服务端项目跳过了 UI 选项，不会让用户疑惑"为什么要我设计界面"
- **可回溯**：开发完成后用户仍可回到 OD 调整 UI，调整后 EM 生成增量 task 更新前端
- **成本透明**：OD 生成原型不消耗 Claude token（OD 是本地服务），仅在确认后开始开发时才消耗

## 7. 流程控制与稳定性

### 7.1 行为监控

所有 Worker 和 Evaluator 的运行状态必须实时可观测：

- **Worker 监控**：当前任务 ID、开始时间、已耗时、JSONL 事件计数、最后事件类型
- **Evaluator 监控**：当前评估的任务、截图数、image-vision 调用数、当前状态
- **进程健康**：子进程 PID、CPU/内存占用、是否僵死
- **监控面板**：StreamPanel 上方或侧边显示简版状态栏，异常时变色
- **事件日志**：所有 spawn/exit/error/kill 事件记入结构化日志

### 7.2 开发锁 + 需求队列

开发进行中锁定项目，新需求排队，本轮完成后统一处理：

```
用户提需求
  → 如果开发进行中：
    → 需求进入队列（显示 "需求已加入队列 #3"）
    → 队列 UI：右侧面板或 Chat 侧边，可拖拽排序
  → 如果开发未运行：
    → 直接追加 task.json → 启动新一轮

本轮开发完成
  → 自动 flush 队列：
    → 按优先级顺序将积攒需求合并到 task.json
    → 启动新一轮开发
    → 队列清空
```

- **队列可管理**：用户可调整优先级、删除、合并需求
- **队列可见**：开发中显示 "当前开发中 + 3 个需求排队"
- **强制模式**：用户可手动暂停开发、清空队列、或立即停止当前任务插入紧急需求

### 7.3 终止流程完善

用户点「停止」后必须保证：

1. SIGTERM → 等待 5 秒 → SIGKILL（给子进程清理时间）
2. 清理子进程树（包括 spawn 的孙子进程）
3. 保存当前进度（task.json 不丢，passes 状态不丢）
4. StreamPanel 显示 "已停止" + 当前进度摘要
5. 释放文件锁
6. Chat 会话不受影响（Worker 死了 Chat 还在）

### 7.4 并发写保护

task.json 是唯一真相源，任何时刻只能一个进程写入：

- **文件锁**：写入前 `flock` / 创建 `.lock` 文件
- **原子写**：先写 `.tmp` → `fs.rename` → 删除旧文件
- **冲突检测**：读取时检查 lock，等待或报错
- **优先级**：用户手动操作 > Worker 自动操作 > Evaluator

### 7.5 进度日志结构化

`progress.txt` 替换为 `progress.json`：

EasyMint 启动时检测开发环境，在 UI 中展示状态指示灯：

### 7.7 Worker 失败处理

- **自动重试**：同一任务失败后自动重试，上限 3 次
- **失败报告**：3 次后暂停，弹出失败详情（错误日志、失败截图、建议）
- **用户决策**：跳过此任务 / 手动修复 / 让 AI 分析
- **级联处理**：如果后续任务依赖失败的任务，自动标记为 blocked

## 8. 沙箱隔离

### 8.1 问题

Worker 和 Evaluator 需要 `--permission-mode bypassPermissions` 才能自动化运行，但 bypass 之后 CLI 内建的权限弹窗不会触发，CLAUDE.md 的安全约束本质上靠 prompt 遵守，没有物理兜底。

### 8.2 方案

**macOS：sandbox-exec + 动态 .sb 配置**

```bash
# EasyMint 在 spawn 前根据 project.path 动态生成 sandbox 配置
cat > /tmp/easymint-worker-${runId}.sb <<EOF
(version 1)
(deny default)
(allow file-read* file-write* (subpath "${project.path}"))
(allow file-read* (subpath "/opt/homebrew"))
(allow file-read* (subpath "/usr/local"))
(allow file-read* (subpath "/private/var/db/xcode_select_link"))
(allow process*)
(allow sysctl*)
(allow network* (local ip))
EOF

sandbox-exec -f /tmp/easymint-worker-${runId}.sb \
  claude -p "..." --permission-mode bypassPermissions
```

**Windows：AppContainer + Low Integrity Level**

```powershell
# 首次创建隔离容器
New-AppContainerProfile -Name "EasyMintWorker" -DisplayName "EasyMint Worker" ...

# spawn 时限定
Start-Process -AppContainer "EasyMintWorker" -Wait -NoNewWindow claude ...
```

### 8.3 能力白名单

| 操作 | 是否允许 | 原因 |
|------|---------|------|
| 读写目标项目目录 | ✅ | worker 只需操作当前项目 |
| 读 node/npm/claude 工具链 | ✅ | 开发工具链路径 |
| npm install（外网） | ✅ | 依赖下载 |
| Vite dev server localhost | ✅ | 开发预览 |
| curl API 测试 | ✅ | 后端验证 |
| git commit | ✅ | 在项目目录内的 .git 操作 |
| 读 ~/.ssh 私钥 | ❌ | 禁止访问用户凭证 |
| 读 ~/Documents、~/Desktop | ❌ | 禁止访问用户数据 |
| 写 /etc、/System | ❌ | 禁止修改系统 |
| 读写其他项目目录 | ❌ | 项目间物理隔离 |
| 外网 curl（非必要） | ❌ | 默认禁止，按需放开 |

### 8.4 关键设计决策

- **项目不动**：沙箱管进程，不管文件。项目仍在用户选的目录，Worker 被物理锁在该目录内
- **动态配置**：每个 Worker/Evaluator spawn 时根据 project.path 动态生成临时沙箱配置
- **平台适配**：macOS sandbox-exec（内建），Windows AppContainer（内建），跨平台无需额外依赖
- **最坏情况**：bypassPermissions 模式下 Claude 即便完全失控，也只能毁掉当前项目，碰不到用户的其他数据
