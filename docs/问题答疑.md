# 用户引导与自动化 — 问题答疑

> 2026-06-10 基于 EasyMint 当前代码分析得出。

---

## 一、背景

用户（Amon）的核心想法：让不懂编程的人也能轻松做出自己的软件。担心引导不到位、用户不按流程操作、自动化不完善。以下为系统分析后发现的问题及用户的逐一答复。

---

## 二、问题清单与答复

### 问题 1：「确认开发」按钮触发方式

**现状**：`ChatPanel.tsx:462` 靠字符串 `lastAiText.includes("确认开发")` 检测。Mint 换种说法按钮就不出现。

**建议方案**：改为 state.json 驱动。Mint 初始化完成后调 `project:writeState({ readyForDev: true })`，前端 `refreshAll` 读到该字段后显示按钮。不再依赖字符串匹配。

**用户答复**：✅ 同意。改成 Mint 主动触发，不用字符串匹配。

---

### 问题 2：空聊天区域引导不足

**现状**：`ChatPanel.tsx:527` 只有一句话「开始对话，让 Mint 帮你开发项目。」。

**用户答复**：✅ 同意适当丰富，但不要太多内容。

---

### 问题 3：技术术语对非技术用户不友好

**现状**：Step 4 表单要求填技术偏好，QuickPrompts 中有"代码审查""技术咨询"等术语。

**用户答复**：Step 4 有 Mint 推荐按钮，不懂直接点就行。术语后续做新手教程引导时统一考虑。

---

### 问题 4-6：Onboarding 跳过、QuickPrompts 入口、功能导览

**现状**：Onboarding 可跳过无后续入口、QuickPrompts 藏在气泡图标里、无 UI 功能导览。

**用户答复**：不讨论，属于锦上添花，后续做一个新手教程引导统一解决。

---

### 问题 7：表单 Step 2-5 不强制填写

**现状**：`canNext()` 只在 Step 1 做了校验，Step 2-5 可以不填直接下一步。

**用户答复**：不填的话默认 Mint 去分析决定。但用户不可能看都不看就下一步，问题不大。

---

### 问题 8：init.sh 执行失败后的行为

**现状**：`buildInitInstruction` 中写了「失败→修改后重试，最多 3 次」，但 3 次后无定义行为。

**用户答复**：Mint 能看到 bash 输出，自己能决定干什么。不应该有需要用户自己执行的脚本，如果有就是超出软件控制范围的，用户可自行寻求帮助。

---

### 问题 9：App 重开后进度倒退

**现状**：如果 Mint 没写 state.json，refreshAll 降级到文件检测，可能退回到 requirements 阶段。

**用户答复**：本质上是需要 Mint 有更强的决策权、监控权、解释权。最理想情况是 Mint 能直接操控软件内的组件——这个需要好好研究。

---

### 问题 10：用户删除/移动项目文件

**现状**：state.json 路径失效时 `catch { /* ignore */ }` 静默吞错。

**用户答复**：用户不会随意删文件，出了问题自己负责。软件面向小白但不面向傻子。

---

### 问题 11：Builder 失败后的升级机制（escalation）

**现状**：Builder 失败 3 次 → 写 `escalation.json` → Mint 读取。但 escalation UI 尚未实现，用户看不到升级信息和操作选项。

**类型定义**（`agent-templates.ts:148`）：
```typescript
interface Escalation {
  type: "escalation";
  from: string;        // Builder 还是 Evaluator
  taskId: string;
  reason: string;
  details: string;
  options: string[];   // ["重试", "跳过", "人工介入"]
  timestamp: number;
}
```

**用户答复**：忘了 escalation 是什么。需确认是否需要做。

**状态**：⚠️ 待确认。此功能在需求跟进文档中列为「高优先级未实现」。

## 三、新提出的方案（待确认）

### 3.1 Mint 操控软件组件

用户希望 Mint 不仅是聊天 AI，还能直接驱动 UI——弹出确认框、高亮按钮、切换面板等。

当前 Mint 只能用 SDK 工具（Read/Write/Edit/Task/MCP）+ IPC（project:writeState），无法操控前端。

**可选方案**：

| 方案 | 做法 | 复杂度 |
|------|------|--------|
| **A. state.json 驱动** | Mint 写 state.json 中的 `uiAction` 字段，前端 refreshAll 时解析并执行 | 低 |
| **B. IPC 通道** | 新增 `agent:uiAction` 通道，Mint 通过 SDK tool 调 IPC | 中 |
| **C. 自定义 MCP Tool** | 给 Mint 注册一个 `easymint_ui_action` MCP tool，直接操作前端 | 中高 |

**状态**：⏳ 待用户确认用哪种方案，或者先记下来后面再说。

## 五、用户的核心诉求（备忘）

1. **Mint 需要有更强的决策权、监控权、解释权**——不只是聊天，要能操控软件
2. **面向小白但不面向傻子**——不过度防御恶意/无意义的用户操作
3. **Mint 自己处理错误**——能看到 bash 输出，自己决策
4. **自动化流程中的人为断点要可靠**——按钮不能靠字符串匹配
5. **新手教程引导后续统一做**——不在本次范围
