# Ask Runtime First

## 原则

**反复修补复杂方案时，退一步问：运行时本身是不是已经解决了这个问题？**

当你在一个外部方案上不断应对边缘情况（进程管理、跨平台、权限、竞态），而这些问题恰好是你所用运行时/框架的设计领域——你的方案很可能本身就是多余的。

## 触发信号

以下任一出现就该触发：
- 同一个 bug 修了两轮还没根除
- 方案里有 `pkill`、`sleep N`、`rm -rf` 中的两个以上
- 你在应对竞态条件、进程生命周期、平台差异——而这些正是运行时管理的领域
- 你在协调「A 停了才能做 B，做完才能启动 A」这类时序——框架通常有专门的 API

## 做法

1. **列出方案中依赖的外部命令/脚本**（bash/python/powershell/curl 等）
2. **逐个问**：运行时/标准库有没有等价 API？（文件操作、进程管理、网络请求、系统对话框）
3. **如果有**——直接用，删掉整个脚本层
4. **如果没有**——保留脚本，但把它当成最后一招，不是首选

## 案例：EasyMint 项目重命名

需求：关闭应用 → 复制目录 → 删旧目录 → 重启。

方案演进：
1. bash 脚本 pkill + rsync + rm + open → pgrep 匹配自己死循环
2. Terminal.app 前台执行 → 跨平台不可用
3. bash 打扫脚本 spawn 后台 → 同样问题

每一步都在给 bash 方案打补丁。

最终方案：`fs.cpSync` + `app.relaunch()` + `app.quit()`。零外部命令，Node.js 标准库 + Electron API 全包。

## 适用范围

这个原则不限于 Electron。React/Vue/Node.js/Python/Go——任何有完整标准库或框架能力的环境，shell out 之前都先问运行时。

## 总结

复杂的脚本方案是坏味道。每当你觉得「再加一个 `sleep` 就好了」「再绕一个进程检查就行了」——停下来，退一步，找运行时原语。
