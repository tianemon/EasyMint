# AORD 需求拆解原则速查

> 来源：`docs/AI驱动开发需求设计原则.md`（AORD v1.0）
> 本文件是精炼版，进入深度拆解模式时加载。

## 11 条原则

### 1. 描述工作流，不描述功能
❌ "需要登录、搜索、收藏"
✅ "用户进入 → 搜索产品 → 查看详情 → 加入对比 → 收藏"

### 2. 从用户目标开始
先写"用户想完成什么"，再拆解为支持流程。

### 3. 树状结构
系统 → 模块 → 页面 → 组件 → 交互 → 接口。
逐层展开，不跳跃。

### 4. 每个节点必须有输入和输出
```
节点名
  输入: ...
  处理: ...
  输出: ...
  异常: ...
```

### 5. 每个功能定义状态变化
```
状态: 未收藏
动作: 点击收藏 → 结果: 已收藏
动作: 取消收藏 → 结果: 未收藏
```

### 6. UI 从工作流推导
先设计流程，页面从流程中自然产生，不凭空添加。

### 7. 功能必须 CRUD 穷举
每个功能都检查：创建/读取/更新/删除。
没写的 Agent 经常不做。

### 8. 显式写出异常流程
```
正常: 密码正确 → 进入系统
异常1: 密码错误
异常2: 账号不存在
异常3: 账号冻结
```

### 9. 页面、数据、API 同步设计
每个功能节点下都写 UI / Data / API 三部分。

### 10. 先全局后局部
永远遵循：系统 → 模块 → 页面 → 组件。不跳跃到细节。

### 11. 全链路数据溯源
关键数据的全局流转路径用三段式标注：
```
起点: 数据从哪里读取
↓
中转: 在哪个介质中持有
↓
终点: 最终写入哪里
```

## 推荐最终结构

```
0. 项目目标
1. 用户角色
2. 用户工作流
3. 功能树
4. 页面树
5. 页面交互
6. 状态机
7. 数据模型
8. API 设计
9. 权限设计
10. 异常流程
11. 测试场景
```

## 从需求到执行

每个叶子节点分配编号（如 `1.1`, `2.3`），Agent 按序执行。
失败时精确定位到编号回滚。
