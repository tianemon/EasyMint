# Web 页面验证工作流

## 何时激活

当用户消息中包含以下任一模式时，**立即**激活此流程：

- "测试一下"、"验证一下"、"确认一下"、"帮我测测"（指向 Web 页面时）
- "开发完了"、"做完了"、"改好了"（关联到前端文件时）
- 用户给出 URL 并要求检查
- git diff 或最近编辑涉及 HTML/CSS/JS/TSX/Vue/Svelte 等前端文件

## 验证流程（严格按顺序，不可跳过）

### 步骤 1：确认服务运行

检查项目是否已有开发服务器在跑。如果没有，启动它（`npm run dev`、`python -m http.server` 等）。

### 步骤 2：打开页面

使用 `mcp__playwright__browser_navigate` 打开目标页面。

### 步骤 3：截图

使用 `mcp__playwright__browser_take_screenshot` 截取页面。**必须**截至少两张：
- 页面初始全貌
- 关键操作后的状态（点击按钮后、提交表单后等）

### 步骤 4：识图验证

**必须**将截图传给视觉模型分析。单张用 `mcp__image-vision__describe_image`，多张用 `mcp__image-vision__describe_images` 并发。

**禁止**截图后自己看图判断——你的视觉能力不可靠，AI 识图结果才是证据。
**禁止**只看 `browser_snapshot` 就下结论——DOM 结构 ≠ 视觉呈现。

### 步骤 5：逐项对比

识图返回后，**必须**逐项回答：

| 检查项 | 预期 | 实际 | 结果 |
|--------|------|------|------|
| 页面标题/关键文案 | | | |
| 核心交互元素可见 | | | |
| 数据正确展示 | | | |
| 无布局错乱/遮挡 | | | |
| 操作后状态正确 | | | |

### 步骤 6：迭代修正

如果发现问题，修正代码后重新从步骤 2 开始。发现的问题 → 修正 → 重新截图验证 → 直到通过。

## 边界测试（UI 修改时必做）

- 空状态页面
- 错误状态/异常数据
- 不同视口尺寸（320px、768px、1440px）
- 操作后状态变化是否正确

## 禁止项

- ❌ 截图后自己看图判断 → **必须**传识图工具
- ❌ 只看 `browser_snapshot` → 看不到视觉 bug
- ❌ 声称"完成"却没有截图 + 识图验证
- ❌ 只截一张图 → 关键状态都要截
